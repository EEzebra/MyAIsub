## Key Questions
1. 入口文件夹（entry/）具体包含哪些类型的入口文件？（环境配置、依赖安装、初始化脚本等）
2. 智能体之间是否需要通信协议或共享状态机制？
3. rules 中的 AI 规则与项目规则的优先级如何？
4. hooks 的触发条件是基于事件还是状态变化？
5. spec 中的场景流程是否需要模板化？

## Decisions Made
| Decision | Rationale |
|----------|-----------|
| agents/ 存放智能体定义 | 按角色划分：计划、编码、测试、架构师、项目维护 |
| 计划智能体为指挥中枢 | 统筹任务全生命周期，调度其他智能体协作 |
| 编码智能体强制架构审核 | 任何架构修改不得绕过架构师智能体审核流程 |
| 测试智能体闭环反馈 | 测试问题反馈给编码智能体修复后回归验证，确保质量 |
| 架构师智能体为架构守门人 | 所有架构修改必须经过架构师智能体审核，防止架构失控 |
| 架构变更分级处理 | 低风险变更记录即可，高风险变更需用户确认，平衡效率与安全 |
| 项目维护智能体驱动体验优化 | 日志管理+Token统计+偏好追踪+规则提取+会话评分 |
| commons/ 集中存放命令 | 所有常用命令脚本统一管理，便于复用和索引 |
| docs/ 作为知识库 | 包含入口文件夹和 source.map，承载全部领域知识 |
| hooks/ 归档条件脚本 | 特定条件下触发的命令脚本集中管理 |
| rules/ 分层规则体系 | 全局规则 > 项目规则 > 场景规则，所有项目必须遵守 |
| spec/ 存放场景流程 | 特定场景下创建任务使用的流程文件，支持模板化扩展 |
