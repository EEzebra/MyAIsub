### 架构师智能体（architect）详细需求

**核心职责：**
- 明确并维护 MyAI 的架构定义
- 监督架构的合理性与完整性
- 严格审核其他 agent 对 MyAI 架构的所有调整行为

**架构审核流程（强制性）：**
```
其他 Agent 发起架构变更
  → 调用架构师智能体审核
    → 低风险变更（命名变更、单个文件层级调整）
      → 记录变更日志，放行
    → 高风险变更（调整文件层次结构、删除/移动目录、新增顶级模块等）
      → 暂停执行，向用户确认
      → 用户批准 → 记录变更日志，放行
      → 用户拒绝 → 撤销变更请求
```

**风险评估标准：**
| 风险等级 | 变更类型示例 | 处理方式 |
|----------|-------------|----------|
| 低风险 | 变量/函数命名变更 | 记录变更即可 |
| 低风险 | 单个文件内的结构调整 | 记录变更即可 |
| 低风险 | 单个文件层级位置微调 | 记录变更即可 |
| 高风险 | 跨目录的文件移动 | 需用户确认 |
| 高风险 | 目录结构的增删改 | 需用户确认 |
| 高风险 | 新增/移除模块或组件 | 需用户确认 |

### 计划智能体（planner）详细需求

**核心目标：** 任务指挥中枢，统筹任务全生命周期

**职责 1 — 任务接收与理解**
- 接收用户任务描述，分析意图和目标
- 明确任务边界和约束条件
- 识别任务涉及的知识领域和所需技能

**职责 2 — 任务拆解与规划**
- 将复杂任务拆解为 3~8 个可执行阶段
- 每个阶段定义明确的输入、输出和验收标准
- 创建 task_plan.md 作为任务蓝图
- 维护 findings.md 记录调研发现
- 维护 progress.md 追踪进度

**职责 3 — 依赖分析**
- 识别子任务之间的前置依赖关系
- 确定并行和串行执行顺序
- 标记关键路径上的任务

**职责 4 — 智能体调度**
- 根据子任务类型分派给合适的智能体：
  - 编码任务 → coder
  - 测试任务 → tester
  - 架构决策 → architect
  - 日常维护 → maintainer
- 协调多智能体协作流程

**职责 5 — 进度跟踪与决策**
- 持续追踪各阶段完成状态（pending → in_progress → complete）
- 在关键决策节点重新审视任务计划
- 遇到阻塞时调整计划，遵循 3-Strike 错误协议

### 编码智能体（coder）详细需求

**核心目标：** 代码执行引擎，高质量完成编码任务

**职责 1 — 代码编写与修改**
- 根据计划智能体分配的任务编写代码
- 优先编辑现有文件，保持改动最小化
- 确保代码可立即运行，包含必要的 import 和依赖

**职责 2 — 规则遵守**
- 严格遵循 rules/ 中的 AI 规则和项目规则
- 遵循编码风格、命名约定、安全规范
- 不输出多余的解释性注释，除非用户要求

**职责 3 — 架构约束**
- 任何涉及架构的修改必须先调用架构师智能体审核
- 审核通过后方可执行，审核未通过需按反馈调整
- 不得绕过架构审核流程

**职责 4 — 技能调用**
- 按需调用 skills/ 下的技能辅助工作
- 生成文档类内容时使用对应技能（docx/pdf/xlsx/pptx 等）
- 开发新技能时遵循 skill-creator 规范

**职责 5 — 增量实现与错误处理**
- 采用增量方式实现，避免大范围重写
- 引入 linter 错误时主动修复，不超过 3 次重试
- 超过 3 次失败则上报计划智能体协调

### 测试智能体（tester）详细需求

**核心目标：** 质量守卫者，确保交付物质量

**职责 1 — 测试用例设计**
- 根据需求和任务计划编写测试用例
- 覆盖正常流程、边界条件和异常场景
- 为每个阶段定义明确的验收标准

**职责 2 — 功能验证**
- 执行测试用例，验证功能是否满足预期
- 检查文件内容、目录结构、代码逻辑的正确性
- 验证文档与实现的一致性

**职责 3 — 问题报告与反馈**
- 记录测试中发现的问题，描述复现步骤
- 将问题反馈给编码智能体修复
- 跟踪问题修复进度

**职责 4 — 回归验证**
- 修复完成后重新执行相关测试
- 确认问题已解决且无新问题引入
- 对通过的测试标记为通过状态

**职责 5 — 结果记录**
- 将测试结果记录到 progress.md 的 Test Results 中
- 统计通过率，作为项目维护智能体评分的数据来源

### 智能体协作流程

```
用户发起任务
  → 计划智能体（planner）：接收任务、拆解规划、创建规划文件
    → 调度各智能体执行子任务
      → 编码智能体（coder）：编写代码
        → 架构变更？→ 架构师智能体（architect）：审核变更
          → 低风险 → 放行
          → 高风险 → 用户确认
      → 测试智能体（tester）：验证成果
        → 问题？→ 回到编码智能体修复
  → 项目维护智能体（maintainer）：日志记录、Token统计、评分
    → 评分 → 调整方向 → 用户确认
  → 完成
```

### 项目维护智能体（maintainer）详细需求
**核心目标：** 提升用户使用体验

**职责 1 — 对话日志管理**
- 用户触发或自动触发时，记录对话原始日志到临时目录
- 压缩日志：只保留「用户输入」和「正确结果」，去掉中间过程
- 日志总量上限 50MB，超出后循环覆盖（最旧日志优先淘汰）
- 日志具有衰减性：越早的日志权重越低，逐步被覆盖

**职责 2 — Token 消耗统计**
- 统计每次对话消耗的 token 数量
- 归档到统计记录文件中

**职责 3 — 任务与偏好追踪**
- 记录对话中与用户完成的任务
- 记录用户的反对意见和补充意见
- 总结用户偏好，记录到用户权重偏好文件
- 记录对话轮次和正确性数量

**职责 4 — 通用规则提取**
- 从对话中总结通用规则
- 将规则记录到对应智能体的通用规则中

**职责 5 — 会话评分与方向调整**
- 汇总以上 4 点数据，对本次会话触发评分
- 对比上一次评分，分析调整方向
- 将评分结果和调整建议与用户确认

## Technical Decisions
| Decision | Rationale |
|----------|-----------|
| 8 阶段搭建 | 按目录逐个建设，由基础到高级渐进推进 |
| 规划文件放 docs/ | docs 是知识库，规划文件属于领域知识的一部分 |
| source.map 作为核心索引 | MyAI 的全局地图文件，便于快速定位项目结构 |
| 架构审核为强制流程 | 任何 agent 修改架构前必须调用架构师智能体审核 |
| 架构变更分级处理 | 低风险记录、高风险确认，兼顾效率与安全性 |
| 日志压缩+循环覆盖 | 去掉中间过程只保留结果，50MB上限循环覆盖，控制存储成本 |
| 多维度会话评分 | 综合日志、Token、任务、偏好、规则 5 个维度评分，数据驱动优化 |
| 计划智能体为指挥中枢 | 统筹任务全生命周期，调度其他智能体协作 |
| 编码智能体强制遵守架构审核 | 任何架构修改不得绕过架构师智能体审核流程 |
| 测试智能体闭环反馈 | 测试问题反馈给编码智能体修复后回归验证，确保质量 |
| 5 智能体协作链路 | planner→coder/architect→tester→maintainer，各司其职 |
