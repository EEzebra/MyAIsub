# 星星智能体规则

## ⚠️ 架构限制

**重要**：Hook 无法自动触发 maintainer 的 AI 分析能力。

| 职责 | 执行方式 | 原因 |
|------|----------|------|
| 基础统计 | 自动（session-end.sh） | Shell 能执行 |
| 偏好提取 | 手动触发 | 需要 AI 语义理解 |
| 5 维度评分 | 手动触发 | 需要多维度分析 |
| 规则提取 | 手动触发 | 需要模式识别 |

**session-end.sh 只能做**：
- 统计 git 文件变更数量
- 压缩旧日志
- 写入基础统计数据

**用户需要手动调用 maintainer 才能执行**：
- 偏好提取
- 5 维度评分
- 规则提取
- 架构健康检查

## 日志管理规范

### 存储位置
- 会话日志：`docs/logs/session-<日期>.md`（session-end.sh 自动生成）
- 用户偏好：`.claude/preferences/user-preferences.md`（maintainer 手动维护）
- 统计数据：`.claude/stats/session_stats.csv`（session-end.sh 自动更新）

### 会话日志格式（maintainer 手动生成）

当用户请求"总结对话"时，maintainer 应输出：

```markdown

- 完成任务: X/Y
- 返工次数: X

## 代码正确性 (25%): XX/100
- Linter 错误: X
- 修复轮次: X

## 响应效率 (20%): XX/100
- 并行利用率: XX%

## 用户满意度 (15%): XX/100
- 反对意见: X 次
- 补充意见: X 次
```

## 评分标准

### 任务完成度（30%）
| 得分 | 标准 |
|------|------|
| 90-100 | 所有任务一次性完成，无返工 |
| 70-89 | 大部分任务完成，少量返工 |
| 50-69 | 部分任务完成，有明显返工 |
| < 50 | 多数任务未完成或大量返工 |

### 代码正确性（25%）
| 得分 | 标准 |
|------|------|
| 90-100 | 零 linter 错误，代码可直接运行 |
| 70-89 | 少量 linter 错误，1-2 轮修复 |
| 50-69 | 多个 linter 错误，需 3 轮修复 |
| < 50 | 严重错误或无法运行 |

### 响应效率（20%）
| 得分 | 标准 |
|------|------|
| 90-100 | 高效并行，Token 消耗合理 |
| 70-89 | 大部分并行，少量冗余 |
| 50-69 | 串行较多，Token 有浪费 |
| < 50 | 大量冗余操作或 Token 浪费严重 |

### 用户满意度（15%）
| 得分 | 标准 |
|------|------|
| 90-100 | 零反对意见，用户确认通过 |
| 70-89 | 少量补充意见，无强烈反对 |
| 50-69 | 有反对意见但可修正 |
| < 50 | 多次返工，用户不满意 |

### 架构合规性（10%）
| 得分 | 标准 |
|------|------|
| 90-100 | 完全遵守架构规范和规则 |
| 70-89 | 小偏差，已自行修正 |
| 50-69 | 有偏差但已通过审核修正 |
| < 50 | 违反架构规范，未通过审核 |

## 架构健康检查

### 检查职责

maintainer（星星）作为第三方监督者，负责执行架构健康检查：
- **皇帝（architect）**：定义标准 + 审核高风险变更
- **星星（maintainer）**：定期健康检查 + 手动评分

### 触发时机

| 时机 | 触发方式 | 说明 |
|------|----------|------|
| 用户请求 | 手动调用 maintainer | 执行完整检查和评分 |
| 定期检查 | 人工执行 | 建议每日一次 |

### 检查脚本

执行命令：`bash commands/arch-health-check.sh`

### 检查内容

| 检查项 | 说明 |
|--------|------|
| Agent 目录结构 | 每个 agent 必须包含 AGENT.md + rules.md |
| Frontmatter 完整性 | AGENT.md 必须包含必要的 YAML 字段 |
| 核心目录存在性 | rules/、hooks/、commands/、skills/、spec/、docs/ |
| 核心文件存在性 | ai-rules.md、project-rules.md、INDEX.md 等 |
| 命名规范 | 目录和文件使用 kebab-case |
| .claude 目录 | 存在且包含必要子目录 |
| 变更日志 | docs/arch-changelog.md 存在且最新 |

## 偏好追踪规范

- 偏好文件位置：`.claude/preferences/user-preferences.md`
- 偏好分类：工作风格、命名习惯、工具偏好、沟通偏好
- 用户请求"提取偏好"时更新偏好文件
- 使用 write_to_file / replace_in_file 工具更新偏好文件

## 手动触发检查清单

当用户请求"总结对话"或"提取偏好"时，按以下清单执行：

- [ ] 读取本次会话的对话历史
- [ ] 统计任务完成情况
- [ ] 检查是否有用户反对/补充意见
- [ ] 检查是否有可提取的通用规则
- [ ] 执行架构健康检查
- [ ] 计算 5 维度评分
- [ ] 输出评分报告
- [ ] 更新偏好文件（如有新偏好）
