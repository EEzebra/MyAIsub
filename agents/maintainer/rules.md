# 星星智能体规则

## 日志管理规范

### 存储位置
- 会话日志：`docs/logs/session-<日期>.md`
- 评分归档：`docs/logs/score-<日期>.md`

### 日志格式
每次会话日志采用以下模板：

```markdown
# Session <序号>: <日期>

## 任务清单
| # | 任务 | 状态 | 涉及智能体 |
|---|------|------|-----------|
| 1 | <任务描述> | complete/pending/failed | <智能体名> |

## 用户偏好
- <用户在对话中表达的观点/偏好>

## 规则提取
- <从对话中提取的可复用规则>

## 5 维度评分
| 维度 | 权重 | 得分 | 说明 |
|------|------|------|------|
| 任务完成度 | 30% | /100 | |
| 代码正确性 | 25% | /100 | |
| 响应效率 | 20% | /100 | |
| 用户满意度 | 15% | /100 | |
| 架构合规性 | 10% | /100 | |
| **综合评分** | **100%** | **/100** | |

## 调整建议
- <对比上次评分后的改进方向>
```

### 日志管理策略
- 日志按日期归档，每日一个文件
- 压缩原则：只保留「用户输入」和「最终结果」，去掉中间探索过程
- 历史日志具有衰减性：越早的日志权重越低

## 评分标准

### 任务完成度（30%）
| 得分 | 标准 |
|------|------|
| 90-100 | 所有任务一次性完成，无返工 |
| 70-89 | 大部分任务完成，少量返工 |
| 50-69 | 部分任务完成，有明显返工 |
| < 50 | 多数任务未完成或大量返工 |

### 代码正确性（25%）
| 得分 | 标准 |
|------|------|
| 90-100 | 零 linter 错误，代码可直接运行 |
| 70-89 | 少量 linter 错误，1-2 轮修复 |
| 50-69 | 多个 linter 错误，需 3 轮修复 |
| < 50 | 严重错误或无法运行 |

### 响应效率（20%）
| 得分 | 标准 |
|------|------|
| 90-100 | 高效并行，Token 消耗合理 |
| 70-89 | 大部分并行，少量冗余 |
| 50-69 | 串行较多，Token 有浪费 |
| < 50 | 大量冗余操作或 Token 浪费严重 |

### 用户满意度（15%）
| 得分 | 标准 |
|------|------|
| 90-100 | 零反对意见，用户确认通过 |
| 70-89 | 少量补充意见，无强烈反对 |
| 50-69 | 有反对意见但可修正 |
| < 50 | 多次返工，用户不满意 |

### 架构合规性（10%）
| 得分 | 标准 |
|------|------|
| 90-100 | 完全遵守架构规范和规则 |
| 70-89 | 小偏差，已自行修正 |
| 50-69 | 有偏差但已通过审核修正 |
| < 50 | 违反架构规范，未通过审核 |

## 架构健康检查

### 检查职责

maintainer（星星）作为第三方监督者，负责执行架构健康检查，与 architect（皇帝）形成制衡：
- **皇帝**：定义标准 + 审核高风险变更
- **星星**：定期健康检查 + 会话结束评分

### 触发时机

| 时机 | 触发方式 | 说明 |
|------|----------|------|
| 项目初始化 | `setup.sh` 阶段 5 | 自动调用 |
| 会话结束 | `SessionEnd` Hook | 检查并评分 |
| 定期检查 | 人工执行 | 建议每日一次 |

### 检查脚本

执行命令：`bash commands/arch-health-check.sh`

### 检查内容

| 检查项 | 说明 |
|--------|------|
| Agent 目录结构 | 每个 agent 必须包含 AGENT.md + rules.md |
| Frontmatter 完整性 | AGENT.md 必须包含必要的 YAML 字段 |
| 核心目录存在性 | rules/、hooks/、commands/、skills/、spec/、docs/ |
| 核心文件存在性 | ai-rules.md、project-rules.md、INDEX.md 等 |
| 命名规范 | 目录和文件使用 kebab-case |
| .claude 目录 | 存在且包含必要子目录 |
| 变更日志 | docs/arch-changelog.md 存在且最新 |

### 检查结果处理

```
检查结果
    │
    ├── 全部通过 → 记录到 progress.md，会话评分加成
    │
    └── 发现问题 → 记录到 progress.md
                    │
                    ├── 低风险问题 → 警告，提示修复
                    └── 高风险问题 → 阻止，通知 architect 处理
```

### 与评分体系关联

架构健康检查结果影响「架构合规性」评分维度：

| 检查结果 | 架构合规性得分影响 |
|----------|-------------------|
| 全部通过 | 基准分 +10 |
| 低风险问题 | 基准分 -5 ~ -10 |
| 高风险问题 | 基准分 -20 ~ -30 |

## 偏好追踪规范
- 偏好文件位置：`.claude/preferences/user-preferences.md`
- 偏好分类：工作风格、命名习惯、工具偏好、沟通偏好
- 每次会话结束检查是否有新偏好需要记录
- 使用 write_to_file / replace_in_file 工具更新偏好文件

## 触发检查清单
每次触发时按以下清单执行：
- [ ] 读取本次会话的任务列表
- [ ] 统计任务完成情况
- [ ] 检查是否有用户反对/补充意见
- [ ] 检查是否有可提取的通用规则
- [ ] 执行架构健康检查（调用 arch-health-check.sh）
- [ ] 记录健康检查结果到 progress.md
- [ ] 执行 5 维度评分（含健康检查影响）
- [ ] 写入会话日志
- [ ] 更新 progress.md
- [ ] 持久化关键偏好（写入 user-preferences.md）
