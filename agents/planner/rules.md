# 魔术师智能体规则

## 任务规划规范

### 任务拆解原则

| 原则 | 说明 |
|------|------|
| 阶段数量 | 3~8 个阶段，避免过细或过粗 |
| 单一职责 | 每个阶段只做一件事 |
| 可验收性 | 每个阶段有明确的完成标准 |
| 依赖明确 | 标注前置依赖关系 |

### task_plan.md 模板

```markdown
# 任务规划

## 任务概述
**目标**: [任务目标描述]
**创建时间**: [日期]
**状态**: planning / in_progress / complete

## Current Phase
[当前阶段名称]

## 阶段列表
| # | 阶段 | 状态 | 依赖 | 负责 Agent |
|---|------|------|------|-----------|
| 1 | [阶段名] | pending/in_progress/complete | - | [agent名] |

## 详细计划

### Phase 1: [阶段名]
**目标**: [阶段目标]
**输入**: [前置条件]
**输出**: [交付物]
**验收标准**: [如何判断完成]

## 风险与假设
- **风险**: [可能的问题]
- **假设**: [已假设的条件]
```

## 智能体调度规则

### 调度决策矩阵

| 任务特征 | 调度目标 | 说明 |
|----------|----------|------|
| 需要写/改代码 | coder | 编码任务 |
| 需要测试验证 | tester | 质量检查 |
| 涉及架构变更 | architect | 必须审核 |
| 需要大范围搜索 | code-explorer (subagent) | 并行探索 |
| 日志/评分/统计 | maintainer | 会话结束触发 |

### 并行调度策略

```
判断是否可并行：
├── 子任务间无依赖 → 可并行
├── 子任务间有依赖 → 串行
└── 部分有依赖 → 有依赖的串行，无依赖的并行
```

**并行调度示例**：
- 同时搜索多个目录 → 并行启动多个 code-explorer
- 代码编写 + 文档更新 → 可并行（不同文件）
- 代码编写 + 测试该代码 → 串行（测试依赖代码）

## 进度跟踪规范

### 状态流转

```
pending → in_progress → complete
    ↑          ↓
    └── blocked ←──┘
```

### 阶段转换条件

| 转换 | 条件 |
|------|------|
| pending → in_progress | 所有前置阶段完成 |
| in_progress → complete | 验收标准全部满足 |
| in_progress → blocked | 遇到阻碍，需外部支持 |
| blocked → in_progress | 阻碍已解决 |

## 确认门设置

### 必设确认门

1. **计划批准门** - task_plan.md 创建后，用户确认
2. **实现开始门** - 实现阶段开始前，确认资源就绪
3. **交付完成门** - 所有阶段完成，用户验收

### 确认门格式

```markdown
## 🚪 确认门

**阶段**: [阶段名]
**待确认内容**: [需要用户确认的内容]
**选项**:
- [ ] 继续执行
- [ ] 调整计划
- [ ] 暂停等待

请确认是否继续？
```

## 3-Strike 错误协议

| 失败次数 | 行为 |
|----------|------|
| 1 次 | 重试，记录原因 |
| 2 次 | 换方法重试，记录原因 |
| 3 次 | 停止，上报用户决策 |

### 上报格式

```markdown
## ⚠️ 需要用户决策

**失败任务**: [任务描述]
**失败次数**: 3 次
**失败原因**:
1. [第一次原因]
2. [第二次原因]
3. [第三次原因]

**建议**:
- 选项 A: [描述]
- 选项 B: [描述]

请选择或提供其他方案。
```
